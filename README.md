# نظام لمسة أزيائي للإنتاج (Lamsa Fashion Production ERP)

نظام ويب متكامل لإدارة إنتاج طلبات متجر **لمسة أزيائي** من لحظة وصول الطلب من سلة حتى التغليف والشحن، مبني بـ Node.js + Express + EJS، وقاعدة بيانات ملف JSON محلي (بدون أي اعتمادات خارجية معقدة، يعمل فوراً على Replit).

## 🚀 التشغيل على Replit (خطوة بخطوة)

1. افتح https://replit.com/~ واضغط **Create Repl** → اختر **Import from GitHub** إن رفعت الملفات لمستودع، أو اختر **Node.js** واسحب/ارفع كل ملفات هذا المشروع مباشرة (فك الضغط أولاً).
2. تأكد أن الملفات في جذر المشروع (بجانبها `package.json` و `server.js`).
3. اضغط زر **Run** — سيقوم Replit تلقائياً بتشغيل `npm install && npm start` (معرف في ملف `.replit`).
4. بعد ثوانٍ سيظهر رابط المعاينة (Webview) — افتحه، وسجّل الدخول.

## 🧩 الوحدات المتوفرة

- **لوحة التحكم**: عدد الطلبات لكل حالة، الطلبات المتأخرة (أكثر من 4 أيام)، إحصائيات يومية وشهرية.
- **الطلبات**: إنشاء يدوي، فلترة، تفاصيل، تغيير الحالة عبر مراحل سير العمل الكاملة، تعيين مشغل.
- **قوائم التجهيز**: فلترة الطلبات حسب المشغل/التاريخ/المنتج/المقاس، وطباعة بطاقات المنتج (بدون اسم العميل) مع باركود.
- **مسح الباركود**: تسجيل خروج للمشغل / دخول من المشغل / دخول المطرز / جاهز للتغليف.
- **المنتجات**: إدارة الكتالوج (مقاسات، ألوان، تطريز).
- **المشاغل**: بيانات المشغل، سجل المهام، حسابات ودفعات.
- **المطرزين**: بيانات المطرز، تحديث القطع المنجزة والأخطاء، حسابات ودفعات.
- **الحسابات**: مواد خام، مصروفات، إجمالي مدفوعات المشاغل والمطرزين.
- **التقارير**: أداء المشاغل، الإنتاج اليومي.
- **المستخدمون**: إدارة الأدوار والصلاحيات (للمدير العام فقط).

## 🔗 ربط متجر سلة (Salla) — خطوات فعلية ومُتحقق منها

⚠️ **مهم:** خاصية "Webhooks" البسيطة الموجودة داخل *إعدادات المتجر* في لوحة تحكم سلة **لا تدعم حالياً أحداث الطلبات** (orders webhooks are hidden for now حسب مركز مساعدة سلة). لذلك الطريقة الصحيحة لاستقبال الطلبات تلقائياً هي إنشاء **"تطبيق خاص" (Private App)** — وهو مجاني وسهل ولا يحتاج مبرمج خارجي أو مراجعة من سلة لأنه لمتجرك فقط.

### خطوات الربط:

1. **أنشئ تطبيقاً خاصاً**: ادخل [portal.salla.partners](https://portal.salla.partners) وسجّل بنفس بيانات دخول متجرك (لتسهيل التوثيق) → "إنشاء تطبيق" → اختر نوع **"تطبيق خاص"** (مجاني، تطبيق واحد خاص لكل تاجر).
2. **فعّل صلاحيات القراءة على الطلبات (Orders scope)** من إعدادات نطاق التطبيق (App Scope)، وتأكد من تفعيل صلاحية **Webhooks** بوضع "قراءة وكتابة".
3. **اربط التطبيق بمتجرك** عبر خطوة "تفويض المتجر" (Store Authorization / OAuth) — هذه الخطوة تُثبّت التطبيق فعلياً على متجرك.
4. من صفحة التطبيق في Partners Portal، ابحث عن قسم **"Webhooks/Notification"** وأضف هذا الرابط:
   ```
   https://<اسم-مشروعك>.<اسم-حسابك>.repl.co/api/webhook/salla
   ```
   ثم اختر الأحداث: `order.created` و `order.status.updated` و `order.cancelled`.
5. سلة ستعرض لك **Secret** خاص بالـ Webhook (يُستخدم للتحقق من صحة الطلبات الواردة). انسخه، وفي Replit افتح أداة **Secrets** (أيقونة القفل 🔒 في الشريط الجانبي) وأضف متغيراً باسم:
   ```
   SALLA_WEBHOOK_SECRET = <القيمة التي نسختها>
   ```
   بدون هذا المتغير، النظام سيستقبل الطلبات دون التحقق من التوقيع (مناسب فقط للتجربة، غير آمن للإنتاج).
6. اختبر الربط: من صفحة التطبيق اضغط "اختبار Webhook" أو أنشئ طلباً تجريبياً في متجرك — سيظهر تلقائياً في قسم "الطلبات" بالنظام خلال ثوانٍ.

### كيف يعمل من الناحية التقنية:
- المسار `/api/webhook/salla` في `server.js` يتحقق من توقيع الطلب (`X-Salla-Signature`) عبر HMAC-SHA256 قبل معالجة أي بيانات — أي طلب بتوقيع غير صحيح يُرفض تلقائياً (401).
- عند استقبال حدث `order.created`، يُنشئ النظام طلباً جديداً بحالة "جديد" مع محاولة استخراج المقاس/اللون/اسم التطريز من خيارات المنتج (Product Options) المرسلة من سلة. بما أن أسماء الخيارات تُكتب من قِبلك عند إنشاء المنتج في سلة (مثلاً "المقاس"، "اللون"، "اسم التطريز")، تأكد من تسميتها بنفس هذه الكلمات أو عدّل الكود في `server.js` (دالة `findOpt`) لمطابقة التسميات الفعلية في متجرك.
- عند إلغاء الطلب من سلة (`order.cancelled` أو تغيير الحالة إلى ملغي)، يُحدَّث الطلب المطابق تلقائياً في النظام.

## 💾 قاعدة البيانات

يستخدم النظام ملف `data/db.json` كقاعدة بيانات (بدون الحاجة لتثبيت PostgreSQL أو أي خادم خارجي)، مما يضمن عمل النظام فوراً على Replit دون إعدادات إضافية. هذا مناسب لحجم الأعمال الحالي (~1000 طلب سنوياً). عند نمو حجم البيانات بشكل كبير مستقبلاً، يمكن تطوير طبقة `lib/db.js` للاتصال بقاعدة PostgreSQL حقيقية دون تغيير بقية الكود (الدوال load/save هي الطبقة الوحيدة التي تحتاج تعديل).

**تنبيه:** خطة Replit المجانية قد تُصفّر الملفات غير المخزنة في Git عند إعادة الإنشاء. لضمان بقاء البيانات، فعّل ميزة **Replit Database/Storage** أو اربط المشروع بمستودع Git واحفظ نسخة احتياطية من `data/db.json` بشكل دوري (مثلاً من صفحة الحسابات أو عبر تنزيل الملف).
